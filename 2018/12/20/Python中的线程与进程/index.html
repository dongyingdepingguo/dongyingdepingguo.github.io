Python中的线程与进程

发表于 2018-12-20 | 分类于 技术篇 | 阅读次数:

进程
进程，按照我的理解就是一个正在进行中的程序，该程序可能正在被执行，也可能处于等待状态，等待被执行。

进程由程序，数据集，进程控制块三部分组成。程序用来描述进程中的哪些功能以及如何完成，数据集是程序执行过程中所使用的资源，进程控制块用来保存程序运行的状态。

由于一个进程往往要同时干多件事，这就需要同时运行对个“子任务”，这些进程内的“子任务”称为线程。

线程
线程又叫轻量级进程，是一个基本的CPU执行单元，也是程序执行过程中的最小单元。

如果你对python的GIL机制有所了解，那么你可能会产生疑问，既然 GIL 保证解释器在同一时间只执行一个线程，那么变量 num 每次只能被一个线程修改，它应该会老老实实的每次减1，为啥还会出现这种情况呢？

事实上 GIL 机制只允许一个时间段内仅有一个线程被执行，过了该时间段，该线程就要释放 GIL 锁，假如这段时间内该线程没有完成数据修改，等到它下次再获得 GIL 去完成上次没完成的任务时，数据可能已经被其他线程改乱了。于是乎，就会出现全局变量 num 被各路线程改的乱七八糟的结果。

如何避免这种情况？很简单，在数据被修改前加锁，修改后释放锁。

Thread-1 执行 f1() ，拿到1锁2锁，再执行 f2() 拿到2锁，Thread-1 sleep 的时候 Thread-2 开始执行 f1() 拿到1锁，接着 Thread-2 想要拿2锁时，2锁在 Thread-1 手里没被释放，然而2锁想要被释放又需要 Thread-2 的1锁，于是两个线程都需要对方手里的锁，但双方又相互攥着不放，于是它们这样相互等待，永远无法执行，这种情况叫做死锁。

所谓死锁：是指两个或两个以上的进程或线程在执行过程中，因争夺资源而造成的一种相互等待的现象，如无外力作用，它们都将无法推进下去。

为了解决这种情况，python 提供了可重入锁 RLock。

RLock（递归锁）
RLock 内部维护着一个 Lock 和一个 counter 变量，counter 记录 acquire 的次数，从而使得资源可以被多次 require。直到一个线程所有的 acquire 都被 release，其他的线程才能获得资源。


值得注意的是获取锁的方式必须是 lock1 = lock2 = threading.RLock() ，如果分开写，与 Lock 没啥区别。

线程间通信——生产者和消费者模型

假如我们的任务中的各个线程是独立，彼此不需要通信，那么这样的任务会简单很多。但实际情况并不是这样，绝大多数都需要线程间相互通信，传输数据。那么如何传输数据呢？简单的做法是设置一个全局变量，某些线程往该变量里写数据，某些线程从变量读数据。在操作该变量是给变量加锁，以保证数据的安全。
