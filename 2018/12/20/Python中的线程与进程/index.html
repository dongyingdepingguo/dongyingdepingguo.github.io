Python中的线程与进程

发表于 2018-12-20 | 分类于 技术篇

线程与进程
进程，按照我的理解就是一个正在进行中的程序，该程序可能正在被执行，也可能处于等待状态，等待被执行。

进程由程序，数据集，进程控制块三部分组成。程序用来描述进程中的哪些功能以及如何完成，数据集是程序执行过程中所使用的资源，进程控制块用来保存程序运行的状态。

由于一个进程往往要同时干多件事，这就需要同时运行对个“子任务”，这些进程内的“子任务”称为线程。

线程又叫轻量级进程，是一个基本的CPU执行单元，也是程序执行过程中的最小单元。

如果你对python的GIL机制有所了解，那么你可能会产生疑问，既然 GIL 保证解释器在同一时间只执行一个线程，那么变量 num 每次只能被一个线程修改，它应该会老老实实的每次减1，为啥还会出现这种情况呢？

事实上 GIL 机制只允许一个时间段内仅有一个线程被执行，过了该时间段，该线程就要释放 GIL 锁，假如这段时间内该线程没有完成数据修改，等到它下次再获得 GIL 去完成上次没完成的任务时，数据可能已经被其他线程改乱了。于是乎，就会出现全局变量 num 被各路线程改的乱七八糟的结果。

Thread-1 执行 f1() ，拿到1锁2锁，再执行 f2() 拿到2锁，Thread-1 sleep 的时候 Thread-2 开始执行 f1() 拿到1锁，接着 Thread-2 想要拿2锁时，2锁在 Thread-1 手里没被释放，然而2锁想要被释放又需要 Thread-2 的1锁，于是两个线程都需要对方手里的锁，但双方又相互攥着不放，于是它们这样相互等待，永远无法执行，这种情况叫做死锁。

所谓死锁：是指两个或两个以上的进程或线程在执行过程中，因争夺资源而造成的一种相互等待的现象，如无外力作用，它们都将无法推进下去。

为了解决这种情况，python 提供了可重入锁 RLock。

RLock（递归锁）
RLock 内部维护着一个 Lock 和一个 counter 变量，counter 记录 acquire 的次数，从而使得资源可以被多次 require。直到一个线程所有的 acquire 都被 release，其他的线程才能获得资源。


值得注意的是获取锁的方式必须是 lock1 = lock2 = threading.RLock() ，如果分开写，与 Lock 没啥区别。

线程间通信——生产者和消费者模型
假如我们的任务中的各个线程是独立，彼此不需要通信，那么这样的任务会简单很多。但实际情况并不是这样，绝大多数都需要线程间相互通信，传输数据。那么如何传输数据呢？简单的做法是设置一个全局变量，某些线程往该变量里写数据，某些线程从变量读数据。在操作该变量是给变量加锁，以保证数据的安全。除此之外，我们还有更好的解决方法——使用 queue 模块。

通过 queue 模块创建一个消息队列，一个线程作为生产者，将通信内容放入队列，一个线程作为消费者，将通信内容从队列中取出，这就是生产者和消费者模型。有了该模型，生产者和消费者不需要直接通信，而是借助阻塞队列，阻塞队列相当于一个缓冲区，平衡生产者和消费者的处理能力。queue 模块进行了较好的封装，保证了在放入和取出时的数据安全。


队列中的常用方法有：
Queue.qsize() : 返回队列大小，但在 Mac OS 上没法运行。
Queue.empty() : 如果队列为空，返回 True，否则 False 。
Queue.full() : 如果队列满的，返回 True，否则 False 。
Queue.get([block[, timeout]]) : 获取队列内容，timeout 为等待时间，如果超过等待时间没有可获取的内容，抛出异常。
Queue.get_nowait() : 无等待直接获取，如果没有内容抛出异常。
Queue.put(item[, block[, timeout]]) : 向队列中放入内容。
Queue.put_nowait() : 相当于 Queue.put(item, False) 。

进程池–Pool
Pool 可以提供指定数量的进程，供用户调用，当有新的请求提交到 pool 中时，如果池还没有满，那么就会创建一个新的进程用来执行该请求，但如果池中的进程数已经达到规定最大值，那么该请求就会等待，直到池中有进程结束。


可在 Pool() 中添加数字设置进程数量，默认为 CPU 核数。
apply_async() 是以非阻塞的方式添加任务，即多个任务可以同时进行。
此外，还有阻塞的方式 apply()。

以阻塞方式添加任务需要等上一个执行完毕，下一个才开始执行，这与单进程没啥区别了。
任务添加完毕后需要 close() 进程池，表示不再接收新任务，使用 join() 阻塞主进程，使子进程全部执行完毕后再运行主进程。
